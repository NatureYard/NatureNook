# 萌宠乐园管理系统开发文档

## 1. 项目概述

### 1.1 项目背景

萌宠乐园业务同时覆盖消费者购票入园、宠物寄养、美容洗护、商品零售、园区物资管理、会员经营和门店现场履约。现阶段需要建设一套统一的管理系统，打通总部、门店、消费者与闸机设备之间的数据链路，降低人工管理成本，提升服务效率，并对票证滥用、人工放行寻租、物资损耗失控等风险建立可审计、可预警的系统能力。

### 1.2 建设目标

- 建设统一的连锁多门店经营管理平台。
- 支持消费者小程序完成购票、预约、会员服务和入园准备。
- 支持商户移动端完成现场核销、寄养、美容、物资领用和异常处理。
- 支持人脸识别闸机接入，实现入园核验、复入控制和人工兜底。
- 支持月卡、季卡、年卡等长期权益产品管理。
- 支持园区内部饲料、清洁用品、耗材等运营物资全流程管理。
- 建立覆盖订单、通行、人工放行、敏感物资操作的审计和风控体系。

### 1.3 一期范围

一期交付以下端与模块：

- Web 管理后台
- 微信消费者小程序
- 商户侧移动端
- 闸机设备接入与门店本地适配服务
- 会员、宠物档案、预约购票、卡种、寄养、美容、零售、库存、物资、报表、风控

### 1.4 非目标

- 暂不建设独立消费者 App。
- 暂不支持多闸机厂商并行适配，仅保留扩展接口。
- 暂不集成复杂实名平台或医疗系统深度能力。
- 暂不拆分微服务，优先单体架构交付。

## 3. 业务范围与模块设计

### 3.1 组织与门店管理

- 支持总部、区域、门店三级视角。
- 支持门店营业时间、服务能力、闸机设备、仓库信息维护。
- 支持员工账号、岗位、数据权限、敏感操作权限配置。

### 3.2 会员中心

- 会员档案、等级、标签、储值余额、积分、优惠券。
- 消费记录、卡种持有记录、售后记录。
- 黑名单和风控关注会员管理。

### 3.3 宠物档案

- 宠物基础信息、品种、性别、生日、体重、照片。
- 主人绑定关系。
- 免疫信息、过敏史、喂养偏好、健康备注。
- 寄养和美容历史服务记录。

### 3.4 门票与服务预约

- 门票按门店、日期、时段、票种销售。
- 服务预约支持寄养、美容洗护等业务。
- 预约支持改约、取消、超容量拦截、退款联动。

### 3.5 卡种权益管理

- 支持月卡、季卡、年卡。
- 支持开卡、续费、冻结、作废、到期提醒。
- 支持限制适用门店、适用品类、有效期。
- 卡种持有人必须实名并绑定人脸使用。

### 3.6 寄养管理

- 入住登记、笼位分配、计划离园时间。
- 喂养记录、清洁记录、异常事件记录。
- 关联物资消耗与服务收费。
- 出园结算、超时费用、附加服务费用。

### 3.7 美容洗护管理

- 服务项目、价格、时长、适用品种。
- 技师分配、开工、完工、现场照片。
- 关联耗材消耗、附加项目收费。

### 3.8 商品零售与收银

- 零售商品主数据、SKU、价格、促销。
- 现场扫码销售、订单支付、退款。
- 零售库存和内部物资库存分账管理。

### 3.9 园区物资与饲料管理

- 物资分类：饲料、零食、洗护用品、清洁用品、寄养耗材、急救常备品、办公物料。
- 物资属性：品牌、规格、单位、保质期、批次、适用宠物类型、是否可售、是否内部领用。
- 业务动作：采购入库、调拨、领用、退库、盘点、报损、过期预警、安全库存预警。
- 消耗登记：按门店、宠物、服务单、员工、时间维度记录。
- 成本归集：寄养单、美容单、门店运营成本。

### 3.10 财务与报表

- 营收报表：门票、寄养、美容、商品、卡种收入。
- 运营报表：会员增长、复购、预约转化、闸机通行、异常放行。
- 库存报表：零售库存、运营物资、损耗、过期、周转。
- 风控报表：人工放行、异常通行、敏感物资操作。

## 4. 核心业务流程

### 4.1 消费者购票入园流程

1. 消费者在小程序选择门店、日期、票种或服务并支付。
2. 系统生成订单、预约记录和待核验入园资格。
3. 消费者在小程序完成或提交人脸预录入。
4. 到店后进行人脸确认。
5. 闸机识别成功后放行，并写入首次入园记录。
6. 系统生成当日通行资格，支持当天多次出入。

当前实现中，消费者单次门票预约支付完成后即创建 `pass_entitlement` 记录；订单中心和商户核销队列直接复用该资格数据。

### 4.2 普通顾客当日复入流程

1. 顾客当天离园后再次返回闸机。
2. 闸机上传人脸识别请求。
3. 系统校验本人、人脸绑定关系、当日通行资格、营业时间、订单状态。
4. 校验通过则再次放行，并追加出入园记录。

### 4.3 卡种客户通行流程

1. 会员购买月卡、季卡或年卡。
2. 首次使用时完成会员实名和人脸绑定。
3. 后续在卡有效期内可多次出入。
4. 系统校验卡状态、适用门店、黑名单状态、营业时段。

### 4.4 人工放行兜底流程

1. 闸机异常、识别失败或特殊客诉触发人工处理。
2. 店员在商户移动端发起人工放行。
3. 系统要求填写原因、关联订单/会员、上传凭证或备注。
4. 有权限的员工可执行放行，系统留审计记录并触发风控规则。

### 4.5 寄养与物资联动流程

1. 前台或员工创建寄养单并分配笼位。
2. 员工按日记录喂养、清洁、健康异常。
3. 每次喂养或清洁领用饲料和耗材时同步登记物资消耗。
4. 出园时系统汇总服务费用与物资成本，形成结算依据。

## 5. 闸机与通行风控专项设计

### 5.1 设计原则

- 人脸识别作为主通行介质。
- 人工放行作为兜底，不作为常规主路径。
- 所有通行行为可追溯、可审计、可预警。
- 消费者票证或卡种与本人绑定，默认不可外借。

### 5.2 通行规则

#### 普通顾客

- 首次入园需具备有效订单或预约。
- 首次成功核验后，生成当日营业时段内有效的通行资格。
- 当天可不限次数出入，次日自动失效。

#### 月卡、季卡、年卡客户

- 在卡有效期内可多次出入。
- 按卡状态校验：有效、冻结、作废、过期。
- 按门店适用范围校验，不默认跨店互通。

#### 拦截规则

- 人脸未绑定或人脸不匹配。
- 订单退款、作废、取消。
- 卡种冻结、作废、过期。
- 黑名单会员。
- 超营业时间或门店不匹配。

### 5.3 防外借设计

- 首次通行完成订单/卡种与人脸标识绑定。
- 后续仅允许绑定人脸通行。
- 更换人脸绑定需要后台受控操作，必须留痕。
- 不接受仅凭截图码、订单号直接重复通行。

### 5.4 防寻租与风控设计

- 人工放行、补录、放行撤销全部留痕。
- 风控指标：
  - 同员工高频人工放行
  - 无有效订单的放行
  - 深夜或低峰时段异常放行
  - 同一会员短时间频繁进出
  - 退款后仍发生通行
- 风控处置：
  - 生成预警事件
  - 门店店长复核
  - 总部后台查看预警报表

当前实现已覆盖的事件样例：

- 黑名单会员尝试入园
- 无有效资格尝试入园
- 订单状态不合法仍尝试核销
- 人工放行登记成功

### 5.5 门店本地适配服务

- 负责对接单一闸机厂商 SDK 或本地接口。
- 接收云端下发的校验结果和通行策略。
- 上报设备心跳、识别请求、放行结果、设备异常。
- 支持短时离线缓存当日已授权通行资格，但离线放行记录必须回传。

### 9.3 关键实体说明

#### `pass_entitlement`

- 用途：表示某会员在某门店的有效通行资格。
- 关键字段：`member_id`、`store_id`、`source_type`、`source_id`、`valid_from`、`valid_to`、`status`
- 规则：
  - 普通票首入园成功后生成当日资格。
  - 卡种可生成跨天长期资格。

#### `entry_exit_record`

- 用途：记录每次入园或出园事件。
- 关键字段：`member_id`、`device_id`、`direction`、`pass_result`、`risk_flag`、`occurred_at`

#### `material_consumption_record`

- 用途：记录运营物资消耗。
- 关键字段：`store_id`、`material_item_id`、`biz_type`、`biz_id`、`pet_id`、`staff_id`、`quantity`
- 业务类型：寄养喂养、美容耗材、清洁消耗、运营耗材

## 10. 接口设计

- 管理后台 API：配置、管理、审批、报表、风控
- 消费者小程序 API：购票、预约、会员、宠物、卡种、售后
- 商户移动端 API：核销、寄养、美容、物资、异常处理
- 门店设备适配 API：通行校验、设备心跳、事件上报

### 10.2 关键接口

#### 消费者端

- `GET /api/c-app/orders`
- `POST /api/c-app/reservations`
- `GET /api/c-app/passes`
- `POST /api/c-app/cards/purchase`
- `POST /api/c-app/faces/pre-enroll`

## 11. 非功能与安全合规要求

### 11.1 性能要求

- 核销与通行校验接口平均响应时间小于 500ms。
- 高峰期支持多门店并发预约和闸机事件上传。
- 关键查询支持按门店、时间、状态高频检索。

### 11.2 安全要求

- 人脸数据仅用于入园核验。
- 敏感信息加密存储和脱敏展示。
- 管理后台、商户移动端关键操作保留完整审计。
- 接口调用需要签名、Token 或设备认证机制。

### 11.3 可用性要求

- 闸机离线时支持短时兜底。
- 本地适配服务恢复后自动补传事件。
- 关键任务失败需具备重试和告警机制。

## 15. UI/UX 设计规范

### 15.1 规范来源

本项目 UI/UX 规范以以下业界权威资料作为基线，并结合微信小程序与移动端业务场景做本地化约束：

- Nielsen Norman Group：`10 Usability Heuristics for User Interface Design`
- Material Design 3 Foundations：accessibility、content design、design tokens、interaction states、layout
- Apple Human Interface Guidelines：移动端界面中的 clarity、feedback、hierarchy、task focus
- W3C WCAG 2.2 Quick Reference：对比度、文本缩放、可操作性、标签与错误提示

本节不是对外部资料的逐条翻译，而是本项目研发、设计、验收阶段必须遵守的实现要求。

### 15.2 总体原则

- 以任务完成为中心，而不是以页面装饰为中心。每个页面首屏必须明确告诉用户“现在可以做什么”。
- 优先使用用户语言，不使用内部术语、表名或研发视角命名。
- 保持同类组件在不同页面中的样式、命名、状态和交互一致。
- 用界面降低记忆负担：让用户“看见下一步”，而不是自己回忆流程。
- 保持审美克制。视觉表现必须服务于购票、预约、核销、履约、售后等核心任务。

### 15.3 信息层级要求

#### 页面层级

- 每个页面必须至少包含：页面主题、当前状态、主要动作、辅助解释。
- 首屏只能有一个主动作。次级动作数量不宜超过两个。
- 重要信息顺序固定为：结果状态、关键对象、时间地点、补充说明、次级操作。

#### 文案层级

- 页面主标题必须说明页面目标，而不是模块名称堆砌。
- 区块标题必须让用户知道该区块能完成什么，例如“确认到店信息”“当前有效凭证”。
- 辅助文案必须用于解释状态、风险、下一步动作，不能重复标题原意。

#### 字号层级

- 小程序正文默认不小于 `26rpx`。
- 辅助标签、状态胶囊、次级说明默认不小于 `24rpx`。
- 区块标题建议使用 `36rpx` 左右统一收口。
- 页面主标题与关键数值必须显著高于正文，保证首屏 3 秒内可识别重点。
- 禁止同一页面出现大量无规律字号并列；新增页面必须沿用既有字号梯度。

### 15.4 状态反馈规范

#### 可见性

- 加载中、已完成、待处理、失败、已禁用、需补充信息等状态必须显式展示。
- 用户触发操作后，界面必须在合理时间内给出反馈，不能无反馈等待。
- 成功状态需说明结果，失败状态需说明原因与下一步建议。

#### 错误处理

- 错误提示必须使用自然语言，不得直接暴露技术异常或接口字段名。
- 表单错误必须指出具体字段和修正方式。
- 对金额、退款、预约提交、人工放行等高成本动作，应优先预防错误，再展示错误提示。

#### 状态文案

- 状态文案必须短、直接、可操作，例如“当前没有可用宠物档案，请先完善宠物信息”。
- 禁止仅用颜色表达状态，颜色必须配合文字、图标或位置变化共同传达含义。

### 15.5 导航与流程规范

- 重复出现的导航位置和顺序必须保持一致。
- 同类动作在不同页面必须使用一致命名，例如“去预约”“查看凭证”不能在相同语义下频繁变体。
- 多步骤流程必须显示当前步骤、下一步和完成结果。
- 流程中禁止自动改变上下文且不告知用户，例如切换选项后直接跳页。
- 用户必须能看见退出路径、返回路径或兜底路径，尤其在支付、预约、核销和人工放行场景中。

### 15.6 表单与输入规范

- 每个输入项必须有明确标签，不能只依赖占位文案表达含义。
- 已经提供过的信息不得要求用户重复输入；应优先复用会员、宠物、门店上下文。
- 选择器、日期、时段、门店、宠物等高频输入应优先使用识别式选择，而不是回忆式输入。
- 提交按钮在不可用状态下必须说明原因，例如缺少宠物档案、未选择预约项目。
- 对法律、财务或数据提交类动作，必须在提交前提供确认信息。

### 15.7 可访问性与可操作性规范

- 文本对比度至少满足 WCAG 2.2 AA 的最低要求；正文文本应以 `4.5:1` 为基线，非文本交互元素以 `3:1` 为基线。
- 交互目标应满足 WCAG 2.2 的最小点击区域思路，避免过小的按钮和胶囊标签。
- 不能只通过颜色区分可点击、已选中、失败、警告等状态。
- 文本放大后不能导致信息缺失或功能不可用。
- 标题、标签、按钮文案必须描述主题或目的，避免“点击这里”“更多”等弱语义文案。

### 15.8 视觉系统规范

#### 设计 Token

- 颜色、字号、圆角、阴影、间距必须抽象为可复用 token，不允许在新页面中任意扩散临时值。
- 同一类卡片、胶囊、按钮、提示条必须复用同一组 token。
- 品牌表达可以有情绪化颜色，但必须受统一 token 控制。

#### 配色系统规范

本项目配色规则以 WCAG 2.2、Material Design 3 Color System、Apple Human Interface Guidelines Color 为共同基线，采用“语义角色优先、品牌色克制使用、对比度可验证”的策略。

- 颜色必须先定义语义角色，再定义具体色值。至少区分：`surface`、`surface-variant`、`text-primary`、`text-secondary`、`border`、`primary`、`on-primary`、`success`、`warning`、`error`、`info`、`focus`。
- 禁止直接以“橙色 1”“绿色 2”之类视觉命名驱动实现。代码与设计稿都应优先使用角色名，而不是色相名。
- 主品牌色只用于一个页面中的一个主动作或一个主强调区域，不得在同一首屏把多个按钮、标签、导航项都刷成同一高饱和强调色。
- 主动作优先通过“有色背景 + 高对比前景文字”强调，不优先采用“浅底上彩色文字”或“彩底上低对比白字”。这是 Material 3 与 Apple HIG 都强调的高可读性做法。
- 背景层级必须明确分为至少三级：页面背景、内容容器背景、强调容器背景。禁止仅靠阴影区分层级而让背景颜色全部接近。
- 文本颜色必须至少分为三级：主文本、次文本、弱提示文本。主文本不得使用品牌色，避免把“可读文本”和“交互强调”混为一体。
- 分隔线、描边、禁用态、占位态必须使用独立 token，不得直接复用正文文本色降低透明度来凑合。
- 状态色必须语义稳定：成功始终指向完成、可用、通过；警告始终指向风险提示或待确认；错误始终指向失败、拒绝、异常。禁止同一颜色在不同页面同时代表不同业务含义。
- 不得只用颜色表达状态或可点击性。状态色必须同时配合文字、图标、位置或形状变化。
- 渐变只能作为情绪背景或强调容器使用，渐变上的正文与操作文案必须重新验算对比度，不能默认沿用纯色背景上的文字颜色。
- 半透明蒙层、毛玻璃、悬浮导航等组件必须在其“默认静止状态”验证可读性，不能假设滚动到深色内容上时才看得清。
- 若组件位于复杂背景或彩色插画之上，优先让文字保持单色高对比，并增加承托层；不要在彩色背景上再叠多色文字。
- 浅色主题至少提供默认对比版本；如后续引入深色主题或高对比主题，必须为同一语义角色提供成套变体，不允许只反转背景色而沿用原前景色。

#### 配色可访问性阈值

- 正文文本与背景对比度以 WCAG 2.2 AA 的 `4.5:1` 作为最低基线。
- 大号文本、粗体大标题可降至 `3:1`，但仅限真正的大文本，不得把普通正文通过放大 1-2 个级别规避对比要求。
- 图标、描边、输入框边界、选中态底色、按钮容器等非文本 UI 元素与相邻背景的对比度不得低于 `3:1`。
- 关键操作、金融金额、风险提示、预约状态、核销结果等核心信息，建议优先向 AAA 的 `7:1` 方向靠拢，而不是只满足最低线。
- 任何使用透明度得到的前景色都必须在最终叠加后的实际背景上验证对比度，不能只检查原始十六进制值。

#### 行业通用落地模式

- 采用 Material 3 的角色化配色方式组织 token：强调色负责主动作与选中态，中性色负责背景与文本层级，语义色负责状态反馈。
- 采用 Apple HIG 的动态语义色思想组织前景与背景关系：前景色应根据背景明度切换深浅，而不是在所有浅色和渐变背景上机械使用白字。
- 对导航栏、工具栏、底部栏、悬浮卡片等高频容器，优先保持低色彩噪声和单色高对比文案，避免与页面内容层争抢色彩焦点。
- 品牌色应集中在主 CTA、选中态、关键状态标签和少量品牌识别模块中；非核心文本、说明文案、普通卡片不应大面积使用品牌色。
- 同一页面推荐遵循“1 个主强调色 + 1 组中性色 + 1 组语义状态色”的上限，避免形成多主色竞争。

#### 本项目配色 Token 要求

- `brand-primary`：用于主按钮、当前选中导航、主流程关键动作。
- `brand-secondary`：用于次级强调区块、辅助 CTA、品牌氛围块，但不得抢过 `brand-primary`。
- `surface-base`：页面底色；`surface-card`：普通卡片；`surface-highlight`：Hero、摘要卡、重要信息面板。
- `text-primary`：默认正文和标题；`text-secondary`：辅助说明；`text-tertiary`：弱提示与占位说明。
- `state-success`、`state-warning`、`state-error`、`state-info`：只用于对应状态，不与品牌色混用。
- `border-subtle`、`border-strong`：分别用于低存在感分隔与输入/选中边框。
- `focus-ring`：键盘焦点、选中边界或高可见交互环，不得缺省。

#### 当前消费者小程序样式层约束

- `frontend/customer-mini/app.wxss` 顶部的颜色角色注释块是当前小程序的样式层单一事实来源，新增页面应先补角色名，再落具体样式。
- 共享卡片容器、状态胶囊、主按钮、页面 Hero、tone 背景应优先定义在全局样式层，再由页面局部样式补结构，不要在每个页面重复声明整套颜色。
- 页面局部样式可以声明布局与场景差异，但不应再次复制 `card-shell`、通用 surface、通用 tone 背景等颜色规则。
- 如新增色值无法归入现有角色，必须先在文档中补充语义说明，再进入样式实现。

#### 三端统一 Token 命名约束

详细 token 定义与跨端映射见 `docs/frontend-design-tokens.md`。

- 消费者小程序、商户移动端、管理后台必须共享同一套一级语义角色名：`brand-*`、`text-*`、`surface-*`、`state-*`、`border-*`、`focus-*`、`shadow-*`。
- 允许各端因平台特性增加二级角色，但必须在一级语义下扩展，例如 `surface-highlight-start`、`state-risk-fg`、`shadow-soft`，禁止重新引入无语义命名如 `green-1`、`orange-light`、`cardBg2`。
- 同一语义在三端的视觉值可以做轻微适配，但用途必须保持一致。例如 `brand-secondary` 在三端都表示次级品牌强调或主流程辅助强调，不得一端拿来做正文色、另一端拿来做错误态。
- 新增 token 时优先检查是否已有父级角色可复用；只有在父级语义不足以描述场景时，才允许新增二级角色。
- 共享 token 的建议映射为：`brand-primary` 负责主 CTA，`brand-secondary` 负责流程/运营强调，`text-primary` 和 `text-secondary` 负责信息层级，`surface-card` 和 `surface-muted` 负责容器层级，`state-success/warning/error/info` 负责业务状态。
- 消费者小程序的全局入口为 `frontend/customer-mini/app.wxss`，商户移动端的全局入口为 `frontend/merchant-mobile/App.vue`，管理后台的全局入口为 `frontend/admin-web/src/style.css`。三端新增 token 时应优先在各自入口定义，而不是在页面文件里首次出现。
- 页面文件可以消费 token，但不应成为 token 的主定义位置；若某页面率先出现新 token，后续必须在同一轮改动中提升到对应全局入口。

#### 配色验收清单

- 首屏主标题、辅助说明、主按钮、次级按钮是否都能在 3 秒内清晰区分。
- 是否存在浅底白字、彩底低对比灰字、渐变背景上文字发虚的问题。
- 是否存在同一种颜色既表示“可点击”又表示“普通说明文字”的混用。
- 状态色是否在不同页面保持同义，不会一处表示成功、另一处表示普通强调。
- 悬浮导航、浮层、毛玻璃、半透明提示条在默认停留状态下是否仍清晰可读。
- 在不同亮度设备、模拟器和真机环境下，品牌色是否依然保持识别度而不牺牲文本可读性。

#### 组件一致性

- 页面容器、卡片容器、提示条、按钮、底部导航、状态胶囊必须具有一致的结构语言。
- 同类列表项必须保持相同的信息布局，例如标题、说明、标签、价格、动作的相对位置一致。
- 自定义底部导航、顶部 Hero、卡片摘要等高频模式必须沉淀为复用模式，而不是每页重画。

#### 视觉克制

- 渐变、阴影、装饰背景可以使用，但不能影响正文可读性和主要动作辨识度。
- 不得使用过多无意义装饰元素抢占首屏视觉焦点。
- 所有视觉强化必须回答一个问题：是否帮助用户更快完成任务。

### 15.9 本项目三端落地要求

#### 消费者小程序

- 首页首屏必须优先承载购票预约、凭证查看、会员权益三个高频任务。
- 入园凭证页必须优先显示当前可用凭证，历史订单退居次级层级。
- 预约流程必须持续显示当前步骤、已选服务、不可提交原因。
- 宠物档案页必须明确说明档案会复用于预约与核销流程。

#### 商户移动端

- 核销页必须优先显示放行结果、失败原因、人工兜底入口。
- 对人工放行、报损、无订单补录等敏感操作，必须增加风险提示与二次确认。
- 待办、履约、物资场景必须突出时间、状态、责任人、下一步动作。

#### 管理后台

- 报表与列表页必须优先强化筛选、状态、异常项识别与批量操作效率。
- 高风险配置项必须有说明、默认值、影响范围提示。
- 复杂配置场景优先使用分组、渐进披露和上下文帮助，而不是一次性堆满所有字段。

### 15.10 设计评审与验收清单

设计联调或代码评审时，至少检查以下问题：

- 首屏是否能在 3 秒内看懂当前任务与主动作。
- 页面是否明确展示当前状态、失败原因和下一步。
- 用户是否需要记忆前一步信息才能完成当前操作。
- 同类按钮、标签、标题、卡片在不同页面是否一致。
- 文本、对比度、点击区域是否满足移动端可读与可操作要求。
- 表单是否减少重复输入，并在提交前预防高成本错误。
- 装饰性设计是否干扰核心任务。

### 15.11 当前仓库实施要求

- 新增或重构小程序页面时，必须先对照本节规范检查：任务层级、状态反馈、字号梯度、按钮可操作性。
- 样式实现优先复用全局 token 与共享组件模式，避免新页面重新发明一套字号、圆角和阴影。
- 若新增自定义导航、复杂卡片、流程条或提示条，需在代码评审中说明它解决的具体可用性问题。
- 涉及预约、支付、核销、人工放行、退款等高风险流程的 UI 改动，必须附带手工验证步骤。
