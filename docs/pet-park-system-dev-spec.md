# 萌宠乐园管理系统开发文档

## 1. 项目概述

### 1.1 项目背景
萌宠乐园业务同时覆盖消费者购票入园、宠物寄养、美容洗护、商品零售、园区物资管理、会员经营和门店现场履约。现阶段需要建设一套统一的管理系统，打通总部、门店、消费者与闸机设备之间的数据链路，降低人工管理成本，提升服务效率，并对票证滥用、人工放行寻租、物资损耗失控等风险建立可审计、可预警的系统能力。

### 1.2 建设目标
- 建设统一的连锁多门店经营管理平台。
- 支持消费者小程序完成购票、预约、会员服务和入园准备。
- 支持商户移动端完成现场核销、寄养、美容、物资领用和异常处理。
- 支持人脸识别闸机接入，实现入园核验、复入控制和人工兜底。
- 支持月卡、季卡、年卡等长期权益产品管理。
- 支持园区内部饲料、清洁用品、耗材等运营物资全流程管理。
- 建立覆盖订单、通行、人工放行、敏感物资操作的审计和风控体系。

### 1.3 一期范围
一期交付以下端与模块：
- Web 管理后台
- 微信消费者小程序
- 商户侧移动端
- 闸机设备接入与门店本地适配服务
- 会员、宠物档案、预约购票、卡种、寄养、美容、零售、库存、物资、报表、风控

### 1.4 非目标
- 暂不建设独立消费者 App。
- 暂不支持多闸机厂商并行适配，仅保留扩展接口。
- 暂不集成复杂实名平台或医疗系统深度能力。
- 暂不拆分微服务，优先单体架构交付。

## 3. 业务范围与模块设计

### 3.1 组织与门店管理
- 支持总部、区域、门店三级视角。
- 支持门店营业时间、服务能力、闸机设备、仓库信息维护。
- 支持员工账号、岗位、数据权限、敏感操作权限配置。

### 3.2 会员中心
- 会员档案、等级、标签、储值余额、积分、优惠券。
- 消费记录、卡种持有记录、售后记录。
- 黑名单和风控关注会员管理。

### 3.3 宠物档案
- 宠物基础信息、品种、性别、生日、体重、照片。
- 主人绑定关系。
- 免疫信息、过敏史、喂养偏好、健康备注。
- 寄养和美容历史服务记录。

### 3.4 门票与服务预约
- 门票按门店、日期、时段、票种销售。
- 服务预约支持寄养、美容洗护等业务。
- 预约支持改约、取消、超容量拦截、退款联动。

### 3.5 卡种权益管理
- 支持月卡、季卡、年卡。
- 支持开卡、续费、冻结、作废、到期提醒。
- 支持限制适用门店、适用品类、有效期。
- 卡种持有人必须实名并绑定人脸使用。

### 3.6 寄养管理
- 入住登记、笼位分配、计划离园时间。
- 喂养记录、清洁记录、异常事件记录。
- 关联物资消耗与服务收费。
- 出园结算、超时费用、附加服务费用。

### 3.7 美容洗护管理
- 服务项目、价格、时长、适用品种。
- 技师分配、开工、完工、现场照片。
- 关联耗材消耗、附加项目收费。

### 3.8 商品零售与收银
- 零售商品主数据、SKU、价格、促销。
- 现场扫码销售、订单支付、退款。
- 零售库存和内部物资库存分账管理。

### 3.9 园区物资与饲料管理
- 物资分类：饲料、零食、洗护用品、清洁用品、寄养耗材、急救常备品、办公物料。
- 物资属性：品牌、规格、单位、保质期、批次、适用宠物类型、是否可售、是否内部领用。
- 业务动作：采购入库、调拨、领用、退库、盘点、报损、过期预警、安全库存预警。
- 消耗登记：按门店、宠物、服务单、员工、时间维度记录。
- 成本归集：寄养单、美容单、门店运营成本。

### 3.10 财务与报表
- 营收报表：门票、寄养、美容、商品、卡种收入。
- 运营报表：会员增长、复购、预约转化、闸机通行、异常放行。
- 库存报表：零售库存、运营物资、损耗、过期、周转。
- 风控报表：人工放行、异常通行、敏感物资操作。

## 4. 核心业务流程

### 4.1 消费者购票入园流程
1. 消费者在小程序选择门店、日期、票种或服务并支付。
2. 系统生成订单、预约记录和待核验入园资格。
3. 消费者在小程序完成或提交人脸预录入。
4. 到店后进行人脸确认。
5. 闸机识别成功后放行，并写入首次入园记录。
6. 系统生成当日通行资格，支持当天多次出入。

### 4.2 普通顾客当日复入流程
1. 顾客当天离园后再次返回闸机。
2. 闸机上传人脸识别请求。
3. 系统校验本人、人脸绑定关系、当日通行资格、营业时间、订单状态。
4. 校验通过则再次放行，并追加出入园记录。

### 4.3 卡种客户通行流程
1. 会员购买月卡、季卡或年卡。
2. 首次使用时完成会员实名和人脸绑定。
3. 后续在卡有效期内可多次出入。
4. 系统校验卡状态、适用门店、黑名单状态、营业时段。

### 4.4 人工放行兜底流程
1. 闸机异常、识别失败或特殊客诉触发人工处理。
2. 店员在商户移动端发起人工放行。
3. 系统要求填写原因、关联订单/会员、上传凭证或备注。
4. 有权限的员工可执行放行，系统留审计记录并触发风控规则。

### 4.5 寄养与物资联动流程
1. 前台或员工创建寄养单并分配笼位。
2. 员工按日记录喂养、清洁、健康异常。
3. 每次喂养或清洁领用饲料和耗材时同步登记物资消耗。
4. 出园时系统汇总服务费用与物资成本，形成结算依据。

## 5. 闸机与通行风控专项设计

### 5.1 设计原则
- 人脸识别作为主通行介质。
- 人工放行作为兜底，不作为常规主路径。
- 所有通行行为可追溯、可审计、可预警。
- 消费者票证或卡种与本人绑定，默认不可外借。

### 5.2 通行规则
#### 普通顾客
- 首次入园需具备有效订单或预约。
- 首次成功核验后，生成当日营业时段内有效的通行资格。
- 当天可不限次数出入，次日自动失效。

#### 月卡、季卡、年卡客户
- 在卡有效期内可多次出入。
- 按卡状态校验：有效、冻结、作废、过期。
- 按门店适用范围校验，不默认跨店互通。

#### 拦截规则
- 人脸未绑定或人脸不匹配。
- 订单退款、作废、取消。
- 卡种冻结、作废、过期。
- 黑名单会员。
- 超营业时间或门店不匹配。

### 5.3 防外借设计
- 首次通行完成订单/卡种与人脸标识绑定。
- 后续仅允许绑定人脸通行。
- 更换人脸绑定需要后台受控操作，必须留痕。
- 不接受仅凭截图码、订单号直接重复通行。

### 5.4 防寻租与风控设计
- 人工放行、补录、放行撤销全部留痕。
- 风控指标：
  - 同员工高频人工放行
  - 无有效订单的放行
  - 深夜或低峰时段异常放行
  - 同一会员短时间频繁进出
  - 退款后仍发生通行
- 风控处置：
  - 生成预警事件
  - 门店店长复核
  - 总部后台查看预警报表

### 5.5 门店本地适配服务
- 负责对接单一闸机厂商 SDK 或本地接口。
- 接收云端下发的校验结果和通行策略。
- 上报设备心跳、识别请求、放行结果、设备异常。
- 支持短时离线缓存当日已授权通行资格，但离线放行记录必须回传。

- 门店设备适配 API：通行校验、设备心跳、事件上报

- 风控标记
- 是否需要人工处理

## 11. 非功能与安全合规要求

### 11.1 性能要求
- 核销与通行校验接口平均响应时间小于 500ms。
- 高峰期支持多门店并发预约和闸机事件上传。
- 关键查询支持按门店、时间、状态高频检索。

### 11.2 安全要求
- 人脸数据仅用于入园核验。
- 敏感信息加密存储和脱敏展示。
- 管理后台、商户移动端关键操作保留完整审计。
- 接口调用需要签名、Token 或设备认证机制。

### 11.3 可用性要求
- 闸机离线时支持短时兜底。
- 本地适配服务恢复后自动补传事件。
- 关键任务失败需具备重试和告警机制。

## 12. 测试与验收标准

### 12.1 功能验收
- 会员注册、绑宠、储值、积分累计与抵扣正常。
- 门票与服务预约创建、改约、取消、退款链路闭环。
- 月卡、季卡、年卡开通、冻结、续费、到期处理正确。
- 消费者小程序购票、预约、售后、人脸预录入流程可用。
- 商户移动端核销、寄养、美容、物资领用操作可用。
- 物资入库、调拨、盘点、报损、过期预警可用。

### 12.2 闸机与风控验收
- 普通顾客首入园成功后当天可多次出入。
- 普通顾客次日再次尝试通行被拦截。
- 卡种客户在有效期内多次出入成功。
- 票证外借、人脸不匹配、退款后通行、跨店通行被正确拦截。
- 闸机离线、识别失败、人工放行、补传恢复链路可用。
- 员工高频人工放行、异常物资报损、无订单放行可触发预警。

### 12.3 数据与报表验收
- 通行记录、订单记录、人工放行记录、物资流水一致。
- 总部与门店权限隔离正确。
- 营收、库存、损耗、风控报表口径一致。

## 13. 实施建议

### 13.1 迭代顺序
1. 基础平台：组织、权限、门店、会员、宠物、商品与服务主数据
2. 交易与履约：预约购票、卡种、寄养、美容、收银
3. 闸机能力：人脸绑定、通行资格、设备适配、人工放行
4. 物资管理：仓库、入库、领用、盘点、报损、成本归集
5. 风控报表：预警规则、审计查询、经营报表

### 13.2 联调重点
- 小程序支付与订单状态回写
- 小程序人脸预录入与门店确认
- 商户移动端人工放行与风控事件生成
- 门店本地适配服务与闸机厂商联调
- 物资消耗与寄养、美容业务联动

## 14. 风险与后续规划
- 单一闸机厂商能力不足时，需尽早补齐适配层抽象。
- 人脸识别误判会直接影响现场体验，需在文档中要求人工兜底闭环。
- 物资和零售库存交叉使用时，必须严格区分库存用途和流水类型。
- 二期可考虑增加多厂商闸机接入、员工绩效、供应链采购协同、更多自动化风控规则。
