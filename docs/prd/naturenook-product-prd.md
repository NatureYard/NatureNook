# Product Requirements Document: 萌宠乐园一体化经营平台（基于当前实现）

**Version**: 1.0  
**Date**: 2026-04-07  
**Author**: Codex  
**Document Basis**: `backend/`、`frontend/admin-web/`、`frontend/customer-mini/`、`frontend/merchant-mobile/` 当前实现代码与测试用例

## Executive Summary

萌宠乐园一体化经营平台面向宠物乐园、宠物门店和连锁品牌方，目标是把“会员与宠物档案、预约下单、支付、入园核销、寄养洗护履约、物资损耗、风险留痕”串成一条可运营、可审计、可持续扩展的业务链路。客户购买这套系统，不是为了获得一个静态后台，而是希望把门店现场服务、消费者体验和总部经营管理放到同一套数据体系中。

基于当前已实现代码，本产品已经形成一个可演示、可试运行的 MVP：消费者侧可在微信小程序完成登录、预约、支付确认、订单查看、卡种查看、宠物档案查看；门店侧可在商户移动端完成核销、人工放行登记、寄养记录、美容完工、物资报损；总部侧可在管理后台查看经营概览、风险事件、闸机记录、会员宠物与预约数据，并进行部分录单操作。当前版本已经具备“从预约支付到到店核验再到风险留痕”的主链路，但仍处于偏试运营与开发联调阶段，尚未补齐正式商用所需的微信真实能力、权限体系、审批流和复杂运营配置。

## 1. 产品定位与当前版本边界

### 1.1 产品定位

- 对品牌方/门店客户而言，这是一套覆盖消费者经营、现场履约和基础风控的一体化经营系统。
- 对门店一线员工而言，这是一套面向现场作业的轻量工作台，核心目标是“少切系统、少手工登记、少口头交接”。
- 对宠物主人而言，这是一套围绕预约、到店、服务记录和会员权益的微信服务入口。

### 1.2 当前版本定义

当前代码实现对应的版本定义为：

- 一个包含消费者小程序、商户移动端、管理后台、后端服务和闸机接口的单体式产品雏形。
- 支持单店到轻量连锁场景的数据模型，但多门店运营规则仍以基础结构为主。
- 支持“预约下单 -> 支付确认 -> 生成通行资格 -> 门店核销/闸机校验 -> 风险留痕”的核心闭环。
- 支持寄养、美容、物资管理的基础履约动作，但尚未形成完整的审批、售后、财务和 BI 闭环。

### 1.3 当前不应被误判为已完整交付的能力

- 微信登录与微信支付当前为开发存根流程，不是正式生产可直接上线的微信接入方案。
- 管理后台当前更接近“经营总览 + 录入入口 + 列表看板”，不是完整的总部配置中台。
- 消费者侧当前以查看和下单为主，宠物档案新增/编辑、卡种购买续费、售后退款等能力尚未落地。
- 门店侧已支持人工放行登记，但尚未实现审批流、图片凭证上传和复核闭环。

## 2. 客户角色与核心需求

### 2.1 品牌方 / 门店经营负责人

- **角色目标**：希望用一套系统统一看会员、订单、通行、风控和库存，不再依赖 Excel、纸质登记和微信群沟通。
- **核心痛点**：
  - 消费者预约、到店和门店履约数据割裂。
  - 人工放行和异常处理缺少留痕，难追责。
  - 门店寄养、美容、物资消耗数据难归档，经营复盘依赖人工汇总。
- **当前系统价值**：
  - 可在后台看到经营概览、风险事件、闸机事件、库存预警。
  - 可在同一系统中查看会员、宠物、卡种、预约、寄养、美容等核心记录。

### 2.2 门店前台 / 店员 / 店长

- **角色目标**：希望在门口和服务现场快速完成核销、异常处理、寄养与洗护记录，不耽误接待和服务节奏。
- **核心痛点**：
  - 核销慢、顾客排队、异常订单处理靠口头判断。
  - 寄养与美容服务缺少统一记录入口，交接容易漏信息。
  - 物资领用和报损没有及时登记，导致库存不准。
- **当前系统价值**：
  - 商户移动端提供任务看板、核销队列、人工放行、寄养记录、美容完工、库存查看与报损入口。

### 2.3 宠物主人 / 消费者会员

- **角色目标**：希望在一个微信入口里完成预约、支付、查看订单和了解自己的卡种与宠物信息，到店流程清晰、少等待。
- **核心痛点**：
  - 预约后不知道是否生效，入园凭证不透明。
  - 订单支付中断后容易丢单。
  - 宠物服务信息和会员权益分散在多个渠道。
- **当前系统价值**：
  - 小程序支持首页、服务选择、预约下单、支付确认、订单查看、卡种查看、宠物档案查看和个人信息查看。

## 3. 核心业务场景

### 场景 A：宠物主人预约并支付后，到店使用服务

- 客户希望用户能够在微信小程序中完成服务选择、宠物选择、日期时段选择、支付确认，并在订单页确认“是否已成功下单、是否可到店使用”。
- 当前实现中，用户可选择 `DAY_TICKET`、`GROOMING_PACKAGE`、`BOARDING_DAY` 三类项目，系统先创建 `PENDING_PAY` 订单，再通过预支付和支付确认把订单更新为 `PAID`。
- 对单次门票/服务预约，支付完成后系统会自动生成通行资格，用于后续凭证查询和门店核销。

### 场景 B：门店前台核销到店订单，遇到异常时人工兜底

- 客户希望前台先看到当天待处理订单、当前资格状态和人脸绑定状态，再决定直接放行还是人工复核。
- 当前实现中，商户移动端可查看当日核销队列，显示订单号、会员、宠物、订单类型、预约时段、资格状态、人脸录入状态。
- 核验通过时，系统写入出入园记录；无有效资格、黑名单或订单状态异常时，系统会阻断并记录风险事件。
- 如现场确需放行，店员可登记人工放行，系统同步生成风险事件，便于后续审计。

### 场景 C：寄养与美容服务履约留档

- 客户希望店员在服务过程中能快速补录寄养入住、每日记录、美容单和完工说明，减少服务信息遗失。
- 当前实现中，门店可创建寄养单、记录寄养每日健康/异常备注，创建美容单并登记完工说明。

### 场景 D：经营负责人关注库存和异常风险

- 客户希望知道哪些物资低于安全库存、哪些人工放行或核销异常需要优先处理。
- 当前实现中，后台和商户端都可查看库存；后台可查看风险事件、闸机事件和总体 KPI；商户工作台会聚合核销、寄养、物资和风险待办。

## 5. 当前 MVP 用户故事与验收标准

### Story 1：消费者完成预约并支付

**作为一名** 宠物主人  
**我希望** 在小程序里选择服务、预约日期和宠物后完成支付  
**以便于** 我到店时能直接使用服务，不需要重复沟通

**当前验收标准：**

- [x] 用户可以在小程序中看到单次门票、洗护套餐、寄养预约三类服务。
- [x] 用户提交预约时必须选择属于自己的宠物，否则系统拒绝创建预约。
- [x] 预约创建后系统生成 `PENDING_PAY` 订单。
- [x] 支付确认后订单状态更新为 `PAID`。
- [x] 支付完成后系统自动生成通行资格，供后续核销使用。
- [x] 待支付订单可在订单页再次发起支付。

### Story 2：门店前台完成核销并处理异常

**作为一名** 门店前台  
**我希望** 先核验订单与资格，再决定是否人工放行  
**以便于** 我能快速放行正常用户，同时把异常情况留痕

**当前验收标准：**

- [x] 前台可以查看当天核销队列，并看到会员、宠物、订单、资格、人脸状态。
- [x] 合法订单且存在有效资格时，系统返回允许放行，并记录出入园事件。
- [x] 订单状态异常时，系统返回拦截结果并创建风险事件。
- [x] 无有效资格或黑名单会员时，系统返回拦截结果并记录风险。
- [x] 人工放行登记成功后，系统自动创建风险事件。

### Story 3：门店员工补录寄养与美容履约

**作为一名** 门店服务人员  
**我希望** 在移动端补录寄养和美容服务过程  
**以便于** 门店能形成完整服务记录，交接时不遗漏信息

**当前验收标准：**

- [x] 店员可以创建寄养单并记录笼位、入住和计划离店时间。
- [x] 店员可以为寄养单新增每日记录，填写健康备注和异常说明。
- [x] 店员可以创建美容单，登记预约时间、技师和费用。
- [x] 店员可以为美容单执行完工登记并写入服务备注。

### Story 4：经营负责人查看经营与风险总览

**作为一名** 店长或总部经营负责人  
**我希望** 在后台查看今日经营、风险和库存风险  
**以便于** 我能知道门店是否存在需要优先处理的问题

**当前验收标准：**

- [x] 后台可显示会员总量、活跃卡种、今日预约、今日入园、未结风险事件、低库存物资数量。
- [x] 后台可查看闸机事件、风险事件、库存、会员、宠物、卡种和预约列表。
- [x] 后台可进行会员、预约、寄养单、美容单的基础新增操作。

## 6. 当前版本上线适用性判断

### 6.1 适合的使用场景

- 单店或少量门店的内部试运行。
- 面向投资人、客户或内部团队的产品演示。
- 用于梳理真实门店流程、验证字段设计和后续上线需求。

### 6.2 暂不建议直接按正式商用版本承诺的场景

- 对外宣称已完成正式微信登录与微信支付闭环。
- 对外承诺总部级权限管理、审批流和全量门店经营配置已齐备。
- 对外承诺已经支持完整售后退款、卡种售卖续费、库存入库盘点、财务对账。

### 6.3 关键上线前补齐项

- 真实微信登录与手机号绑定。
- 真实微信支付、异步回调、订单超时关闭。
- 严格鉴权与 token 持久化。
- 人工放行证据上传、审批与处置闭环。
- 更完整的异常提示、日志、监控与权限控制。

## 7. 未来优化方向

### 7.1 P0：从试运行走向可上线版本

- 客户希望会员从进入小程序开始就是真实身份闭环，因此需要补齐微信登录、手机号绑定、实名认证和正式支付回调。
- 客户希望异常订单、人工放行和风险事件能形成闭环，因此需要增加审批、复核、事件关闭和证据材料。
- 客户希望门店员工在不同账号下看到属于自己的数据，因此需要补齐员工登录、门店隔离和角色权限。
- 客户希望支付、核销、设备上报更稳定，因此需要补齐 Redis token、监控告警、错误码治理和日志追踪。

### 7.2 P1：提升门店运营效率

- 客户希望消费者能自助管理宠物档案和会员权益，因此应增加宠物建档编辑、卡种购买续费、订单取消改约、消息通知。
- 客户希望门店现场更少手工录入，因此应增加扫码核销、服务模板、自动带出会员宠物信息、任务联动跳转。
- 客户希望寄养和美容服务更可视化，因此应增加照片上传、服务过程记录、技师排班和顾客评价。
- 客户希望库存更准确，因此应增加入库、调拨、盘点、批次与过期预警。

### 7.3 P2：支持连锁经营和精细化增长

- 客户希望总部做跨店经营分析，因此应增加多门店报表、门店对比、经营趋势和风险趋势分析。
- 客户希望系统不仅管履约，还能带来复购，因此应增加会员标签、优惠券、活动营销、召回触达。
- 客户希望对宠物服务形成长期档案，因此应增加健康档案、喂养偏好、服务偏好和历史消费画像。
- 客户希望系统逐步标准化接入更多硬件，因此应增加多闸机厂商适配、设备状态中心和离线同步能力。

## 8. 建议追踪的产品成功指标

**业务指标**

- 预约支付成功率
- 到店核销成功率
- 人工放行占总放行比例
- 寄养/美容服务记录完整率
- 低库存问题处理及时率
- 风险事件 24 小时闭环率

**体验指标**

- 消费者从选服务到完成支付的平均时长
- 到店核验平均耗时
- 小程序待支付订单二次支付转化率
- 店员日均手工补录次数

**经营指标**

- 活跃会员数和活跃卡种数
- 今日预约到店率
- 服务类订单复购率
- 物资异常损耗率
